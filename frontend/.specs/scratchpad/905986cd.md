## Phase 1: Requirements Discovery

### Task Overview
- Initial User Prompt: "you'll find in frontend directory the simulator task which should take place in simulator menu instead of what's already here." (pointing to frontend/SIMULATOR_TASK.md)
- Current Description: Replace free-form manual duel playground on /simulator with a single-player Yu-Gi-Oh! combo sandbox powered by the real EDOPro rules engine (ocgcore-wasm).
- Task Type: feature
- Complexity: XL (4 pillars, WASM integration, real rules engine, hard feasibility gate, AGPL risk)

### Problem Definition (step by step)

Step 1 — Surface request: Swap the current manual /simulator board for an engine-backed sandbox built per SIMULATOR_TASK.md's 4 Pillars.

Step 2 — What user actually wants: A goldfish/combo tester where card effects, chains, negates, and continuous effects resolve under REAL Yu-Gi-Oh! rules. The current manual board lets you place cards but enforces no rules — so it cannot validate whether a combo actually works (does the negate land? does the chain resolve in the right order? does the continuous effect lock something?). The user wants to rehearse combos and see real interactions.

Step 3 — Business value: TradeMarket is a YGO card marketplace/deck site. A working combo simulator differentiates it, increases engagement/time-on-site, gives deck builders a reason to return, and complements existing deck/card pages. It validates combos players are considering before they buy cards — ties into the marketplace.

Step 4 — Who benefits: Competitive/combo players testing lines; deck builders validating a deck's engine; the business (engagement, differentiation, SEO via a unique interactive tool).

Step 5 — Future/adjacent features: shareable combo replays (the command log makes this natural), combo-of-the-day, linking a simulated line back to deck pages, the existing static combo-index/parser as a suggestion layer. (Most are OUT of scope now.)

Step 6 — Constraints: Engine is AGPL-3.0 (copyleft exposure). Must run client-side only (WASM never at build time; /simulator excluded from SSG). Asset delivery (cards.cdb + ~13k CardScripts) is non-trivial. The mid-duel injection handshake is UNVERIFIED — hence Pillar 0 is a hard gate. Sync vs async (JSPI) browser execution unknown.

Therefore root problem: "The current /simulator cannot tell a player whether a combo actually works because it enforces no rules. Players need a single-player sandbox where real engine rules resolve effects/chains/negates so they can rehearse and validate combos — built on an as-yet-unproven mid-duel card-injection model that must be feasibility-gated before any UI is built."

### Scope
- Included: full 4-pillar feature; Pillar 0 spike as a hard gate; replace manual board; reuse DuelField.vue/CardPicker.vue as visual layer; engine = source of truth; free-entry/effects-enforced boundary; format/banlist as candidate-pool FILTER only; command-replay rewind.
- NOT included: multiplayer/netcode, deckbuilding, banlist ENFORCEMENT, entry-legality enforcement (search cost/filter, ED materials), the static combo-index/parser feature.
- Boundaries: engine enforces what cards DO in play, not HOW they enter.

### Ambiguous areas
- Exact post-injection refresh step (Debug.ReloadFieldEnd vs bare query) — resolved by Pillar 0, intentionally a spike output, not a pre-decided AC.
- Sync vs async build — resolved by Pillar 0.
- Asset delivery (bundle vs R2 fetch) — flagged as NFR/risk, not pre-decided.
- AGPL approach — explicitly "document, do not re-decide".

## Phase 2: Concept Extraction

Step 1 — Actors:
- Combo player / deck builder (single human user; the only actor — no opponent).
- The rules engine (ocgcore-wasm) — autonomous system actor that resolves rules and emits decisions.
- (Implicit) asset infra: cards.cdb + CardScripts source.

Step 2 — Actions/behaviors:
- Initialize a duel (fixed seed).
- Inject a card into ANY zone (hand/field/GY/banished/S-T/deck) = free grab from full pool.
- Summon an ED monster (materials skipped).
- Activate an injected card's effect; engine resolves chains/negates/continuous.
- Respond to engine-prompted decisions (select card/chain/yes-no/position).
- Re-render board from engine state after each step.
- Rewind/step back via command replay; step forward again.
- Filter candidate pool by format/banlist.

Step 3 — Data entities:
- Duel handle + seed.
- Card (passcode/code from cards.cdb), Script (c<code>.lua).
- Board/Field state (zones, cards, positions) read via duelQueryField/duelQueryLocation.
- Pending decision (MSG_SELECT_* with valid options).
- Input log: ordered list of injections + duelSetResponse, with seed (Pillar 3).

Step 4 — Constraints:
- Bootstrap order: constant.lua then utility.lua BEFORE any card.
- Scripts lazy + cached, never preload all.
- No state serialization → rewind must be replay.
- Client-only route; AGPL licensing.

Step 5 — Implicit assumptions:
- A local/static cards.cdb + CardScripts checkout is available (scripts/edopro/ already does this).
- Engine is deterministic given the same seed + same input sequence.
- DuelField.vue's expected shape can be produced from normalized engine state.

Therefore key concepts: single-actor, engine-as-source-of-truth combo sandbox; free entry / enforced-in-play effects; deterministic command-replay model; feasibility-gated on mid-duel injection.

## Phase 3: Requirements Analysis

### Functional Requirements (testable)
- FR0 (GATE): A card injected mid-duel via loadScript+Debug.AddCard is visible in duelQueryField output; the exact post-injection refresh step is documented; injection route recorded; sync-vs-async decision recorded. If injection cannot be confirmed → STOP + escalate.
- FR1: Engine wrapper can init a duel and render an empty board from duelQueryField.
- FR2: duelProcess loop advances and surfaces pending decisions (MSG_SELECT_*) as structured objects with valid options.
- FR3: duelSetResponse correctly answers at least: select card, select chain, yes/no, select position.
- FR4: Board re-renders from engine state after each step; engine (not usePlayground) is source of truth.
- FR5: Player can search any card and inject it into any chosen zone; it appears on the board via engine state.
- FR6: ED picker lists all ED monsters (Fusion/Synchro/Xyz/Link incl. Pendulum hybrids), EXCLUDES plain Pendulum Main-Deck monsters; format filter applies.
- FR7: An injected card's own effect can be activated and resolves under engine rules.
- FR8: Candidate lists are paginated/filterable (some return thousands).
- FR9 (rewind): Replaying the full input log reproduces an identical board (determinism).
- FR10 (rewind): Stepping back N then forward reproduces the same states.
- FR11 (rewind): A step list / undo affordance is driven off the log.

### Non-Functional Requirements (measurable)
- NFR-Perf-Rewind: Rewind of a ~20-action combo completes in well under 1 second.
- NFR-Perf-Load: Engine WASM loads lazily on /simulator entry only; card scripts fetched on demand and cached (never preload ~13k scripts); a card's script loads on first use.
- NFR-ClientOnly: /simulator stays a client-only route — WASM/engine never runs at build; route remains excluded from vite.config.js ssgOptions.includedRoutes; no prerender of board state.
- NFR-AssetDelivery: cards.cdb + CardScripts delivery strategy is documented (bundle vs fetch-on-demand from existing Cloudflare R2), tied into scripts/edopro/ asset prep.
- NFR-Licensing: AGPL-3.0 exposure of ygopro-core/ocgcore is documented (obligations + delivery options) as an acknowledged risk; approach NOT re-decided here.
- NFR-Browser: sync core preferred unless JSPI confirmed available in target browsers (decision recorded in Pillar 0).

### Constraints & Assumptions
- Bootstrap constant.lua+utility.lua before any card.
- Reuse DuelField.vue + CardPicker.vue as visual layer; reuse comboResolver.js for candidate lists; extend api.js for format/banlist filtering.
- Deterministic engine w/ explicit seed.
- Entry legality and ED materials intentionally NOT enforced — ACs must NOT require entry-legality enforcement.

### User Scenarios

Primary Flow (combo rehearsal with real effects):
1. User opens /simulator; engine loads client-side, empty board renders from engine state.
2. User searches a card, drops it into hand/field (injection); it appears via engine state.
3. User summons an ED monster from the ED picker (materials skipped); it appears on the field.
4. User activates an injected card's effect; engine resolves the chain/negate/continuous and prompts decisions.
5. User answers prompts (select card/chain/yes-no/position) via duelSetResponse; board re-renders after each step.
6. User reviews the resolved line; optionally rewinds to a prior step and tries a different line.

Pillar-0 Spike Outcome (gating milestone):
- PASS: card injected mid-duel appears in duelQueryField; refresh step + route + sync/async recorded → proceed to Pillars 1–3.
- FAIL: no clean mid-duel injection → STOP, escalate; "deck is the DB" model is reconsidered before any UI work.

Error / Edge scenarios:
- Engine-load failure (WASM unavailable / browser unsupported / asset fetch fails): /simulator surfaces a clear non-crashing error state; rest of app unaffected.
- Script fetch failure for a specific card: that injection fails gracefully with a user-visible message; the duel remains in a consistent state (no half-injected card).
- Huge candidate list (thousands of results): list is paginated/filterable; UI stays responsive.
- Injection that emits no MSG_*: UI reconciles by re-reading duelQueryField/duelQueryLocation so the card still appears (no silent loss).
- Rewind determinism break (replay yields different board): treated as a defect; determinism check (FR9) must catch it.
